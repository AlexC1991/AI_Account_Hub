# Community API Worker

This staging Worker is the signed ingestion and private storage boundary for AI
Account Hub community telemetry. The desktop application never receives an R2
S3 key, Cloudflare account token, or direct bucket credential.

The configured `TELEMETRY_BUCKET` binding targets `ai-hub-data`. Wrangler uses
local simulated R2 storage by default during `npm run dev`; the account's S3 API
URL is intentionally not used by Worker code.

## Current routes

- `GET /v1/health`: non-sensitive service status.
- `GET /v1/schema`: public client/schema compatibility.
- `GET /v1/community/models`: returns real privacy-thresholded aggregates when
  available, otherwise the explicitly labelled synthetic staging preview.
- `POST /v1/installations`: registers a P-256 public key after proof of possession.
- `POST /v1/submissions`: accepts a signed, allowlisted daily aggregate.
- `DELETE /v1/installations/me`: deletes accepted raw submissions and the
  installation's D1 state after a signed request.

Unsigned writes are rejected. Timestamp and nonce checks prevent replay, D1
uniqueness makes daily submission idempotent, rate-limit bindings provide a
coarse abuse layer, and all R2 object keys are generated by the Worker.

Every accepted payload is copied into private `daily_contributions` rows and
reduced into daily model/reasoning rollups. Publication requires the configured
minimum number of distinct installations for both the cohort and each visible
day. Submission and withdrawal refresh the published object immediately; an
hourly Cron Trigger reconciles rollups after interrupted cross-storage writes.

Staging read endpoint:

```text
https://ai-account-hub-community-api-staging.acca-gaming-productions.workers.dev
```


## Cloudflare staging

Authenticate interactively with `npx wrangler login` or supply a scoped
`CLOUDFLARE_API_TOKEN` through a protected CI environment. Set
`CLOUDFLARE_ACCOUNT_ID` outside the repository when the account cannot be
resolved from Wrangler login. Apply D1 migrations with `--remote`, run
`npm run dry-run`, and deploy only after the signed local flow passes.

The D1 database ID and R2 bucket name in `wrangler.jsonc` are binding metadata,
not credentials. Authentication remains in Wrangler or protected CI secrets.
